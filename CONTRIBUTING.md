# CONTRIBUTING

## Summary

这份文档用于约束当前项目的开发与提交方式，目标是让 UI 快速迭代、服务层扩展和后续真实设备接入保持一致。

## 基本原则

- 业务核心类型优先集中在 `src/core/types.h`
- UI 不直接依赖真实协议细节
- 设备接入统一通过 `IDeviceAdapter`
- 本地存储统一通过 Repository 和 Service
- 小步提交，避免一次混合多类改动

## 目录约定

- `src/app`
  放应用上下文与服务装配
- `src/core`
  放核心类型、公共转换和应用配置
- `src/device`
  放设备接口与适配器实现
- `src/storage`
  放仓储与本地持久化
- `src/ui`
  放主窗口、页面、弹窗和控件
- `scripts`
  放打包与发布脚本

## 开发流程

推荐流程：

1. 明确要改的页面或模块
2. 先确认是否已有可复用类型或控件
3. 修改代码
4. `cmake --build build -j 4`
5. 运行应用并做手工验证
6. 更新必要文档

## UI 改动约定

- 优先保持现有布局结构稳定
- 样式优化尽量和布局调整分开做
- 截图对标时，一轮只处理一个明显问题
- 公共控件的样式尽量抽局部常量，避免每页一套

## 服务与类型改动约定

- 新页面状态或业务字段，优先先定义类型再接 UI
- 不要让 UI 持有协议层格式化逻辑
- 服务层负责流程控制，仓储层负责本地文件

## 发布相关约定

- 正式发布使用 Git tag，格式：`vX.Y.Z`
- 打包脚本保持平台职责单一
- GitHub Actions workflow 改动时，同步更新 `RELEASE.md`

## 文档约定

以下场景应同步更新文档：

- 新增页面或大改页面结构
- 新增核心类型或服务接口
- 修改构建或发布流程
- 增加新的本地依赖

优先更新：

- `README.md`
- `ARCHITECTURE.md`
- `DEVELOPMENT.md`
- `RELEASE.md`
- `UI_GUIDE.md`
- `API_GUIDE.md`
- `TEST_PLAN.md`
