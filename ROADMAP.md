# ROADMAP

## Summary

这份文档用于描述当前项目后续开发路线，目标是把“可运行原型”推进到“可联调、可发布、可维护”的桌面应用版本。

当前路线按 4 个阶段推进：

- `R1` 工程化收口
- `R2` 真实设备接入
- `R3` 业务闭环完善
- `R4` 质量与发布升级

## R1 工程化收口

目标：

- 把当前原型整理成可持续开发的仓库状态

重点事项：

- 提交当前文档和基础仓库文件
- 配置 GitHub remote
- 触发一次真实 GitHub Release
- 清理发布流程中的平台差异问题
- 固化常用开发命令和发布命令

交付标准：

- 仓库结构稳定
- GitHub Actions 可以完成一次成功发布
- 文档与代码状态一致

## R2 真实设备接入

目标：

- 让 `RealDeviceAdapter` 从预留壳层进入可联调状态

重点事项：

- 明确真实协议、寄存器、报文和状态格式
- 为连接、状态、扫描、校正、开关控制补齐真实实现
- 保持 `IDeviceAdapter` 接口不变
- 对接调谐页和监测页的数据回流

交付标准：

- 可以连到真实设备或协议模拟器
- 能获取状态、启动扫描、停止扫描
- UI 层无需为真实协议重写

## R3 业务闭环完善

目标：

- 把页面从“可展示”推进到“可完成主要工作流”

重点事项：

- 扫描设置、系统设置、谱图设置补齐交互细节
- 方法管理页从占位页升级为可用模块
- 数据页完善单帧浏览、筛选、恢复和导出
- 异常路径提示和状态反馈补齐

交付标准：

- 调谐流程可完整跑通
- 监测方法可保存、加载、执行
- 数据保存和重载可闭环

## R4 质量与发布升级

目标：

- 把项目从原型级提升到更稳定的交付级

重点事项：

- 增加基础自动化测试
- 建立 UI 回归检查清单
- 统一页面样式常量和设计约束
- 增加 Windows 安装器或更正式的分发形式
- 评估 macOS codesign / notarization

交付标准：

- 构建、打包、发布更加稳定
- 关键页面变更可回归验证
- 发布产物更接近正式交付

## 里程碑建议

### M1

- 提交当前文档体系
- 打通 GitHub Release

### M2

- 完成真实设备连接与状态读取

### M3

- 完成真实扫描与调谐/监测联动

### M4

- 完成方法管理和数据页闭环

### M5

- 完成测试与正式发布增强

## 当前优先级建议

如果按投入产出比排序，建议优先顺序：

1. `R1` 工程化收口
2. `R2` 真实设备接入
3. `R3` 业务闭环完善
4. `R4` 质量与发布升级
